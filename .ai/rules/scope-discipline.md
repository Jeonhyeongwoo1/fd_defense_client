# Scope Discipline Rules

사용자가 명시한 범위만 작업한다. 범위 외 작업은 사용자 확인 후 진행한다.

## Principles

- 요청을 단어 단위로 읽는다. "Show만" 이라면 Show 흐름만, "팝업을 띄워" 라면 팝업 띄우기까지만.
- "깔끔하다 / 기왕이면 / 미래에 필요할 것 같다" 같은 사유로 작업을 확장하지 않는다.
- 인접한 파일·시스템도 사용자 발화에 명시되지 않았으면 손대지 않는다.
- 사용자가 작업 일부를 거부하면, 그 한 가지 항목만 좁게 해석하지 말고 **전체 범위 인식이 잘못되었을 가능성을 먼저 의심**한다.

## Anti-Patterns

- 사용자가 "popup 클래스 만들어"라고 했는데 구매·sprite·바인딩까지 한 번에 구현
- 사용자가 "버튼 추가"라고 했는데 인접 시스템(Presenter·Service·DI)을 함께 재설계
- 거부된 항목 1개만 되돌리고 나머지 임의 확장 작업은 그대로 두기
- "다음 단계에서 어차피 필요하니까" 같은 추론으로 미리 작업
- 새 파일·시스템·서비스를 사용자 명시 없이 신규 생성

## Required Behavior

- 명시 범위와 실제 작업 범위가 일치하는지 작업 전 1회 확인한다.
- 작업 중 "이것도 같이 하면 좋겠다"는 발견이 생기면 **작업하지 말고 보고**한 뒤 결정을 받는다.
- 거부 신호(취소·"왜 그랬어?"·범위 지적)가 들어오면 **전체 작업 그림을 재검토**하고 사용자에게 옵션을 제시한다.
- 위임(implementer 등) 시에도 명세에 명시 범위만 정확히 전달한다. 위임 명세에 "기왕이면" 류의 추가 작업 포함 금지.

## When in Doubt

- 작업 범위가 모호할 때는 **추측하지 말고 질문**한다. (AskUserQuestion)
- 사용자가 명확히 결정한 항목 외에는 default가 "하지 않음".

## Good

```
사용자: "버튼 클릭 → popup Show 되도록"
→ 작업: 버튼 핸들러 + popup.Show() 호출 라인 추가. 끝.
```

## Bad

```
사용자: "버튼 클릭 → popup Show 되도록"
→ 작업: popup의 모든 SerializeField 정의 + Presenter 신규 + 모델 바인딩 + Service 호출 흐름 + DI 등록 + Addressables + Editor 자동화 스크립트
```
